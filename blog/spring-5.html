스프링 5주차 회고

점진적으로 개선해나간다. 회고를 하면 얻을 수 있는 좋은점입니다. 저는 이번주 어떤 것들을 발견했고, 개선했을까요?

> 발견한 것

계속해서 cucumber를 사용하고 있어요. 이런 테스트 방식을 다른 프로젝트에도 적용해보았고, 나쁘지 않은 것 같아요. describe context it 보다 이게 더 편한 것 같네요. 이게 더 편한 이유는 given(context)를 미리 생각하는게 저에겐 더 익숙한 것 같아요. 말로 표현하는 것도 그게 더 자연스러워요. 이런 방식에서 제가 중요하게 생각하는 부분은 정말 말이 되는 테스트 설명문인데, 더 자연스럽다고 생각하는 방식을 선호하는건 당연한 것 같아요.

회고를 더 자주 보도록 매일 캘린더에 설정한게 유효한 것 같습니다. 실제로 회고를 더 자주 보았어요. 그래서 BDD style을 모든 테스트에 적용시켰죠! 이 방식이 효과적인 것 같습니다. 저는 자주 회고를 하지만, 모든 회고를 기록하지는 않고 잊어버리는 경우도 꽤 있습니다. 이런점을 어떻게 보완할지 더 고민해봐야겠어요.

저는 좋아보이는게 있으면 다른 곳에도 많이 적용하는 것 같아요. 예를들어서 라이브러리를 추가하는 경우에 pr에 설명을 쓰면 좋다는 것들이죠. 슬랙에 글이 올라오자마자 제 나름대로 한다음 공유를 했는데, 이렇게 시도해보는건 좋은 습관인 것 같아요.

저는 여전히 annotation이 불편해요. 적어도 도메인 레이어에 annotation이 들어오는건 최대한 막고싶어요. lombok의 경우도 도메인 레이어에 사용하는건 너무나 거슬리더라구요. annotation이 붙었다고 뭔가 다른 동작을 바로 하진 않지만, 사용하지 않는 경우가 불편하지 않다면 안쓰는 편이 좋은 것 같아요. 물론 트레이드오프가 있겠지만, 적어도 저는 이게 좋아요.

IntelliJ IDEA는 매우 많은걸 자동으로 해주고 똑똑한 도구라서 편하네요. 다른 언어를 사용하는 것과 자바를 쓰는 경험은 이런 지원이 너무 다른 것 같아요. Spring도 비슷해요. 다른 언어는 DI도 직접해줘야하고 미리 정의된 JPA같은 repositroy도 없고 힘든게 많은 것 같아요. 그래서 이런 것들을 자동으로 해준다는게 좋은 것 같네요.

> 개선할 점

커밋 메세지는 여전히 좋지 않습니다. 다음 과제에는 이걸 가장 중요하게 생각해야겠어요. TDD를 하다보면 스텝이 작아지고, 그러면 어디까지 커밋을 해야하는지 애매한 것 같아요. 스텝을 조금 더 키워도 될 것 같네요.

이번주 수요일날 몸이 안좋아서 휴가를 쓰고 이력서를 조금 써봤어요. 아직 개선할게 많지만, 이력서를 쓰지 않았다면 과제에 더 집중할 수 있었겠죠.

이렇게 DDD를 적용한 프로젝트 개발을 페어프로그래밍 해보고싶네요. 같이하면 더 재미있고 많은걸 배울 수 있다고 생각하는데, 어떻게 할지 고민해봐야겠어요.
